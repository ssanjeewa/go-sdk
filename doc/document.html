ZKP Go SDK — Developer Guide

module github.com/ssanjeewa/go-sdk Go 1.22+ Arbitrum Sepolia · chain 421614

The ZKP Go SDK is a client library for the ZKClaimUpload protocol — a request-scoped zero-knowledge authorization system for decentralized file access control on Arbitrum Sepolia. The SDK wraps the ZKP middleware HTTP API with idiomatic Go types, input validation, retry logic, and standalone ECIES crypto helpers.

Architecture overview

text
  ┌──────────────────────────────────────────────────────────────────┐
  │  Your application                                                │
  │                                                                  │
  │   zkp.NewClient(baseURL, WithAPIKey(...))                         │
  │        │                                                         │
  │        ├── IssueCredential  ──►  ZKP Middleware (Go HTTP)        │
  │        ├── GenerateProof    ──►  Groth16 proof (~600 ms)         │
  │        ├── PrepareShare     ──►  Delegate credential             │
  │        └── GetUserPubKey    ──►  On-chain registry lookup        │
  │                                                                  │
  │   zkpcrypto.GenerateKeyPair()   ◄── standalone, no HTTP          │
  │   zkpcrypto.EncryptSecret(...)                                   │
  │   zkpcrypto.DecryptSecret(...)                                   │
  └──────────────────────────────────────────────────────────────────┘
              │                              │
              ▼                              ▼
      ZKP Middleware              Arbitrum Sepolia (chain 421614)
      (issue + prove)             ZKClaimRegistryV3 / ServiceV3/V4

Quick Start

This minimal example checks the middleware health, generates a key pair, issues a credential, and generates a proof — the two most common operations.

go 
package main

import (
    "context"
    "fmt"
    "log"
    "os"

    zkp       "github.com/ssanjeewa/go-sdk"
    zkpcrypto "github.com/ssanjeewa/go-sdk/crypto"
)

func main() {
    ctx := context.Background()

    // ── 1. Create the client ──────────────────────────────────────
    client, err := zkp.NewClient(
        "http://localhost:3002",
        zkp.WithAPIKey(os.Getenv("ZKP_API_KEY")),
    )
    if err != nil { log.Fatal(err) }

    // ── 2. Check middleware health ────────────────────────────────
    health, err := client.Health(ctx)
    if err != nil { log.Fatal(err) }
    fmt.Printf("status=%s chainId=%d\n", health.Status, health.ChainID)

    // ── 3. Generate a secp256k1 key pair ──────────────────────────
    kp, err := zkpcrypto.GenerateKeyPair()
    if err != nil { log.Fatal(err) }
    fmt.Println("privateKey:", kp.PrivateKey) // store securely!
    fmt.Println("publicKey: ", kp.PublicKey)

    // ── 4. Issue a credential ─────────────────────────────────────
    resp, err := client.IssueCredential(ctx, &zkp.IssueCredentialRequest{
        UserAddress:   "0xYourWalletAddress",
        FileID:        "0xYourFileID...32bytes...hex",
        UserPublicKey: kp.PublicKey,
    })
    if err != nil { log.Fatal(err) }

    claimFile := resp.ClaimFile
    fmt.Printf("leafIndex=%d  commitment=%s\n",
        claimFile.LeafIndex, claimFile.Commitment)
    // Submit resp.InsertCalldata on-chain (batchInsertLeaves)

    // ── 5. Later: decrypt and generate a proof ───────────────────
    secret, err := zkpcrypto.DecryptSecret(
        claimFile.EncryptedSecret, kp.PrivateKey,
    )
    if err != nil { log.Fatal(err) }

    proof, err := client.GenerateProof(ctx, &zkp.GenerateProofRequest{
        UserAddress: "0xYourWalletAddress",
        FileID:      claimFile.FileID,
        N:           secret.N.String(),
        S:           secret.S.String(),
        LeafIndex:   claimFile.LeafIndex,
    })
    if err != nil { log.Fatal(err) }

    fmt.Println("nullifier:", proof.RequestNullifier)
    // Submit proof.Calldata on-chain (executeRequest)
}

Flow 1 — File Upload

Before a user can generate ZK proofs for a file, the file must be registered on-chain and a credential must be issued by the middleware. The upload flow has four steps:

  • 1
    Register public key on-chain Call ZKClaimUserRegistry.registerPublicKey(pubKey) with your secp256k1 public key. This is a one-time on-chain transaction per wallet. Required so the middleware can encrypt credentials for you.
  • 2
    Create file record on-chain Call ZKClaimServiceV3.createFile(fileId, metaHash, keyCommit). This registers ownership and the encrypted file key on-chain. Submit as a UserOp via your ERC-4337 smart account.
  • 3
    Issue credential via SDK Call client.IssueCredential(...). The middleware generates the (n, s) secrets, computes the Poseidon commitment, encrypts the secrets with your public key, and returns a ClaimFile and InsertCalldata.
  • 4
    Submit InsertCalldata on-chain Submit InsertCalldata (encodes batchInsertLeaves) as a sponsored UserOp. This inserts your credential commitment into your personal Merkle tree.
go 
import (
    "context"
    "encoding/json"
    "fmt"
    "log"

    zkp       "github.com/ssanjeewa/go-sdk"
    zkpcrypto "github.com/ssanjeewa/go-sdk/crypto"
)

func uploadFile(client *zkp.ZKPClient, userAddr, fileID string) {
    ctx := context.Background()

    // Step 1: Generate (or load) your secp256k1 key pair.
    // The private key must be stored securely — you need it every time
    // you want to download the file or delegate access.
    kp, err := zkpcrypto.GenerateKeyPair()
    if err != nil { log.Fatal(err) }

    // Step 2: Issue credential — middleware generates (n, s), commitment,
    // and encrypts the secrets with kp.PublicKey via ECIES.
    resp, err := client.IssueCredential(ctx, &zkp.IssueCredentialRequest{
        UserAddress:   userAddr,
        FileID:        fileID,
        UserPublicKey: kp.PublicKey,
    })
    if err != nil {
        log.Fatalf("IssueCredential: %v", err)
    }

    // Step 3: Persist the ClaimFile — required for proof generation later.
    claimJSON, _ := json.MarshalIndent(resp.ClaimFile, "", "  ")
    fmt.Println("Save this ClaimFile:", string(claimJSON))

    // Step 4: Submit InsertCalldata on-chain.
    // resp.InsertCalldata encodes batchInsertLeaves([commitment]).
    fmt.Printf("Submit on-chain:\n  to:    %s\n  data:  %s\n  value: %s\n",
        resp.InsertCalldata.To,
        resp.InsertCalldata.Data,
        resp.InsertCalldata.Value,
    )
}
⚠
Store the ClaimFile and private key securely. The EncryptedSecret in the ClaimFile is needed to decrypt (n, s) for proof generation. Without them, the file cannot be accessed. The private key must never be logged or serialized to disk unencrypted.

Flow 2 — Batch Upload

Issue credentials for up to 20 files in a single API call. The middleware uses a single GetNextLeafIndex call (preventing leaf-index collisions) and returns one InsertCalldata covering all N leaves.

go 
func batchUpload(client *zkp.ZKPClient, userAddr string, pubKey string, fileIDs []string) {
    ctx := context.Background()

    // Build per-file request entries (max 20).
    files := make([]zkp.BatchFileRequest, len(fileIDs))
    for i, id := range fileIDs {
        files[i] = zkp.BatchFileRequest{FileID: id}
    }

    resp, err := client.IssueCredentialBatch(ctx, &zkp.BatchIssueCredentialRequest{
        UserAddress:   userAddr,
        UserPublicKey: pubKey,
        Files:         files,
    })
    if err != nil {
        log.Fatalf("IssueCredentialBatch: %v", err)
    }

    // Each Credentials[i].ClaimFile corresponds to fileIDs[i].
    for i, cred := range resp.Credentials {
        fmt.Printf("[%d] leafIndex=%d  fileId=%s\n",
            i, cred.ClaimFile.LeafIndex, cred.ClaimFile.FileID)
    }

    // ONE calldata for all N leaves — submit once via batchInsertLeaves.
    fmt.Println("Single InsertCalldata to submit:", resp.InsertCalldata.To)
}
✓
Batch issuance saves gas and reduces on-chain round-trips. Use it whenever a user uploads multiple files in the same session.

Flow 3 — File Download (Access)

To access a file, the user proves ownership of a credential in their Merkle tree without revealing the secret. The flow has three steps:

  • 1
    Decrypt credential secrets Use zkpcrypto.DecryptSecret with the EncryptedSecret from your ClaimFile and your private key to recover (n, s).
  • 2
    Generate Groth16 proof Call client.GenerateProof(...) with the decrypted (n, s), userAddress, fileId, and leafIndex. The middleware fetches the Merkle path, computes the proof (~600 ms), and returns a Solidity-ready proof + calldata.
  • 3
    Submit executeRequest calldata on-chain Submit Calldata (encodes executeRequest) as a UserOp. The contract verifies the proof, marks the nullifier used, and emits KeyReleased with the encrypted file key.
go 
func downloadFile(client *zkp.ZKPClient, claimFile *zkp.ClaimFile, privKey string) {
    ctx := context.Background()

    // Step 1: Decrypt (n, s) from the ClaimFile.
    // The private key never leaves this function — do NOT log it.
    secret, err := zkpcrypto.DecryptSecret(claimFile.EncryptedSecret, privKey)
    if err != nil {
        log.Fatalf("DecryptSecret: %v", err)
    }

    // Step 2: Generate proof.
    // n and s are validated against the BN254 scalar field before the HTTP
    // call — an out-of-range value returns *ValidationError immediately.
    proof, err := client.GenerateProof(ctx, &zkp.GenerateProofRequest{
        UserAddress: claimFile.UserAddress,
        FileID:      claimFile.FileID,
        N:           secret.N.String(),  // decimal string
        S:           secret.S.String(),  // decimal string
        LeafIndex:   claimFile.LeafIndex,
    })
    if err != nil {
        log.Fatalf("GenerateProof: %v", err)
    }

    fmt.Println("requestNullifier:", proof.RequestNullifier)
    fmt.Println("reqId:            ", proof.ReqID)

    // Step 3: Submit on-chain.
    // proof.Calldata encodes executeRequest(fileId, reqId, rn, a, b, c, signals, "").
    fmt.Printf("Submit:\n  to:   %s\n  data: %s\n",
        proof.Calldata.To,
        proof.Calldata.Data[:20]+"...",
    )
    // After on-chain success, listen for the KeyReleased event to get
    // the encrypted AES file key, then decrypt the file.
}
ℹ
Each call to GenerateProof consumes one reqId counter slot on-chain. Nullifiers are stored permanently — the same (n, scope, reqId) combination cannot be reused. This is by design: it prevents replay attacks while allowing the same credential to be used for multiple requests.

Flow 4 — Share a File

The owner can delegate access to any registered user (grantee) by re-encrypting the credential secrets for them and preparing a share on-chain.

  • 1
    Fetch grantee's public key Call client.GetUserPubKey(ctx, granteeAddress). Returns the 65-byte uncompressed secp256k1 key that the grantee registered on-chain. Returns Registered: false if the grantee has not yet registered.
  • 2
    Decrypt your own secrets and re-encrypt for grantee Decrypt (n, s) from your ClaimFile, then call zkpcrypto.EncryptSecret(granteePubKey, n, s) to produce an ECIES blob only the grantee can decrypt.
  • 3
    Prepare share via SDK Call client.PrepareShare(...). The middleware issues a fresh delegated credential for the grantee, computes a new commitment, and returns InsertCalldata + GrantShareCalldata.
  • 4
    Submit both calldatas on-chain Submit InsertCalldata first (inserts grantee leaf), then GrantShareCalldata (records the share grant). Send the EncryptedCredential to the grantee out-of-band.
go 
func shareFile(
    client      *zkp.ZKPClient,
    claimFile   *zkp.ClaimFile,
    ownerPrivKey string,
    granteeAddr  string,
) {
    ctx := context.Background()

    // Step 1: Fetch grantee's on-chain public key.
    pubKeyResp, err := client.GetUserPubKey(ctx, granteeAddr)
    if err != nil { log.Fatal(err) }
    if !pubKeyResp.Registered {
        log.Fatalf("grantee %s has not registered a public key", granteeAddr)
    }

    // Step 2: Decrypt your own (n, s) and re-encrypt for grantee.
    secret, err := zkpcrypto.DecryptSecret(claimFile.EncryptedSecret, ownerPrivKey)
    if err != nil { log.Fatal(err) }

    encForGrantee, err := zkpcrypto.EncryptSecret(pubKeyResp.PubKey, secret.N, secret.S)
    if err != nil { log.Fatal(err) }

    // Step 3: Prepare the share — middleware issues delegated credential.
    shareResp, err := client.PrepareShare(ctx, &zkp.SharePrepareRequest{
        FileID:                 claimFile.FileID,
        GranteeAddress:         granteeAddr,
        GranteePublicKey:       pubKeyResp.PubKey,
        EncryptedKeyForGrantee: encForGrantee,
    })
    if err != nil { log.Fatal(err) }

    fmt.Printf("grantee leafIndex: %d\n", shareResp.LeafIndex)

    // Step 4: Submit both calldatas on-chain (order matters!).
    // 1. InsertCalldata  → batchInsertLeaves (grantee's Merkle tree)
    // 2. GrantShareCalldata → grantShare (ZKClaimServiceV4)
    fmt.Println("InsertCalldata:",     shareResp.InsertCalldata.To)
    fmt.Println("GrantShareCalldata:", shareResp.GrantShareCalldata.To)

    // Send EncryptedCredential to the grantee through a secure channel.
    fmt.Println("Send to grantee:", shareResp.EncryptedCredential[:20]+"...")
}

Flow 5 — Grantee File Access

A grantee who has received an EncryptedCredential from an owner can list their incoming shares and generate proofs to access the shared files.

go 
func granteeAccess(client *zkp.ZKPClient, granteeAddr, granteePrivKey string) {
    ctx := context.Background()

    // Step 1: List all shares granted to this address.
    sharesResp, err := client.IncomingShares(ctx, granteeAddr)
    if err != nil { log.Fatal(err) }

    for _, share := range sharesResp.Shares {
        if !share.Active {
            continue // share was revoked
        }

        fmt.Printf("Shared file: %s  from: %s\n", share.FileID, share.Owner)

        // Step 2: Decrypt the credential the owner sent out-of-band.
        // share.EncryptedCredential contains ECIES(granteePublicKey, n||s).
        secret, err := zkpcrypto.DecryptSecret(share.EncryptedCredential, granteePrivKey)
        if err != nil {
            fmt.Printf("  cannot decrypt share: %v\n", err)
            continue
        }

        // Step 3: Generate proof (same as owner access flow).
        // LeafIndex comes from the on-chain CredentialInserted event for
        // this share, or from the PrepareShare response given by the owner.
        proof, err := client.GenerateProof(ctx, &zkp.GenerateProofRequest{
            UserAddress: granteeAddr,
            FileID:      share.FileID,
            N:           secret.N.String(),
            S:           secret.S.String(),
            LeafIndex:   0, // set to actual leafIndex from the insert event
        })
        if err != nil {
            fmt.Printf("  GenerateProof: %v\n", err)
            continue
        }

        fmt.Printf("  proof ready, submit: %s\n", proof.Calldata.To)
    }
}
ℹ
IncomingShares replays on-chain ShareGranted and ShareRevoked events to compute the current active state. Only shares where Active: true are usable.

Crypto — Key Management

Import the crypto sub-package standalone — it has no HTTP dependency.

go 
import zkpcrypto "github.com/ssanjeewa/go-sdk/crypto"

// Generate a fresh secp256k1 key pair (uses crypto/rand internally).
kp, err := zkpcrypto.GenerateKeyPair()
// kp.PrivateKey → "0x" + 64 hex chars (32 bytes, zero-padded)
// kp.PublicKey  → "0x04" + 128 hex chars (65 bytes, uncompressed)

// Derive public key from an existing private key.
pubKey, err := zkpcrypto.PrivateKeyToPublicKey("0xabc...private...key")

// Validate a public key format.
ok := zkpcrypto.IsValidPublicKey(pubKey)
// true  → 0x04-prefixed, 65 bytes
// false → compressed, wrong length, or non-hex
FunctionSignatureNotes
GenerateKeyPair () (*KeyPair, error) Uses crypto/rand. Pads private key to 64 hex chars (leading-zero safe).
PrivateKeyToPublicKey (privHex string) (string, error) Returns *CryptoError if key is wrong length or invalid hex.
IsValidPublicKey (pubHex string) bool Returns true only for 0x04-prefixed 65-byte uncompressed keys.

Crypto — ECIES Encryption

ECIES (secp256k1 + AES-256-GCM) is used to encrypt and decrypt the (n, s) credential secrets. The wire format is byte-for-byte identical across the Go middleware, the Go SDK, and the Node.js SDK — credentials encrypted by any implementation can be decrypted by any other.

Wire format

text
  [ ephPubKey: 65 bytes ] [ IV: 12 bytes ] [ ciphertext: 64 bytes ] [ GCM tag: 16 bytes ]
  = 157 bytes total = 0x + 314 hex characters

  Plaintext layout: n (32 bytes, big-endian) || s (32 bytes, big-endian) = 64 bytes
  AES key = SHA-256(ECDH shared secret x-coordinate)
go 
// Encrypt (n, s) for a recipient's public key.
encrypted, err := zkpcrypto.EncryptSecret(recipientPubKey, n, s)
// encrypted → "0x" + 314 hex chars (157 bytes)
// Returns *CryptoError on invalid public key or encryption failure.

// Decrypt a credential secret with the recipient's private key.
secret, err := zkpcrypto.DecryptSecret(encryptedHex, privateKeyHex)
// Returns *CryptoError on wire-too-short, bad prefix, GCM auth failure.
// The private key is NEVER included in error messages (SEC-01).

fmt.Println(secret.N) // *big.Int — BN254 field element
fmt.Println(secret.S) // *big.Int — BN254 field element
⚠
Never log or serialize *DecryptedSecret. The (n, s) values are the plaintext credential secrets. Go does not guarantee memory zeroing — limit the lifetime of DecryptedSecret to the minimum required scope.

Crypto — BN254 Field Validation

All circuit signals (n, s, scope, public signals) must be valid elements of the BN254 scalar field. The SDK validates n and s before every HTTP call to GenerateProof, avoiding a wasted round-trip on out-of-range values.

go 
// The BN254 scalar field modulus (package-level *big.Int).
// All field elements must satisfy: 0 ≤ v < BN254ScalarField
fmt.Println(zkpcrypto.BN254ScalarField)
// 21888242871839275222246405745257275088548364400416034343698204186575808495617

// Check whether a *big.Int is a valid field element.
ok := zkpcrypto.IsValidFieldElement(v) // v != nil && v >= 0 && v < BN254ScalarField

// Parse a decimal string into a validated field element.
v, err := zkpcrypto.ParseFieldElement("42")
// Returns *ValidationError if not a decimal integer or out of range.

Pass ClientOption functions to NewClient to tune behaviour.

go 
client, err := zkp.NewClient(

OptionTypeDefaultDescription

Updating the API key at runtime

go 
// Safe to call from any goroutine — uses atomic.Pointer[string] internally.
// In-flight requests continue with the old key; new requests use the new key.
client.SetAPIKey("new-rotated-key")

Retry behaviour

The SDK automatically retries requests on these conditions:

  • HTTP 500, 502, 503, 504 — server errors
  • HTTP 429 — rate limited. Waits Retry-After header value (seconds) or the computed backoff, whichever is longer.
  • Transport errors — connection refused, timeout, reset

The SDK does not retry on 401, 403, 404, 409, or other 4xx codes — these indicate a client error.

Error Handling

All methods return typed errors. Use errors.As to check the specific type:

go 
import "errors"

_, err := client.IssueCredential(ctx, req)
if err != nil {
    var authErr       *zkp.AuthError
    var rateLimitErr  *zkp.RateLimitError
    var notFoundErr   *zkp.NotFoundError
    var treeFullErr   *zkp.TreeFullError
    var networkErr    *zkp.NetworkError
    var validationErr *zkp.ValidationError
    var serverErr     *zkp.ServerError

    switch {
    case errors.As(err, &authErr):
        // API key missing or invalid
        fmt.Println("check your ZKP_API_KEY")

    case errors.As(err, &rateLimitErr):
        // Too many requests — back off
        fmt.Printf("rate limited, retry after %dms\n", rateLimitErr.RetryAfterMs)

    case errors.As(err, &notFoundErr):
        // File not registered on-chain (createFile not yet submitted)
        fmt.Println("file not found — is it created on-chain?")

    case errors.As(err, &treeFullErr):
        // User Merkle tree at capacity (2^24 leaves)
        fmt.Println("tree full — wallet rotation required")

    case errors.As(err, &networkErr):
        // Transport error — middleware unreachable
        fmt.Printf("transport: %v\n", errors.Unwrap(networkErr))

    case errors.As(err, &validationErr):
        // Bad input caught client-side before HTTP call
        fmt.Printf("bad request: %v\n", validationErr)

    case errors.As(err, &serverErr):
        fmt.Printf("server error %d\n", serverErr.StatusCode)

    default:
        fmt.Printf("unexpected: %v\n", err)
    }
}

API Methods

Health no auth GET

Health(ctx context.Context) (*HealthResponse, error)

Returns current service status and chain ID from GET /v1/health. Does not require an API key.

Metrics no auth GET

Metrics(ctx context.Context) (string, error)

Returns raw Prometheus metrics text from GET /metrics. Useful for monitoring proof latency, worker pool utilisation, and sync lag.

IssueCredential auth POST

IssueCredential(ctx, *IssueCredentialRequest) (*IssueCredentialResponse, error)

Issues a ZK credential for one (userAddress, fileId) pair. Validates inputs before the HTTP call. Errors: *NotFoundError (file not on-chain), *TreeFullError (depth-24 capacity reached).

FieldTypeRequiredDescription
UserAddressstring✓0x-prefixed 20-byte Ethereum address
FileIDstring✓0x-prefixed 32-byte hex file identifier
UserPublicKeystring✓0x04-prefixed 65-byte uncompressed secp256k1 public key
Secretstring—Optional decimal bigint — dev/test only

IssueCredentialBatch auth POST

IssueCredentialBatch(ctx, *BatchIssueCredentialRequest) (*BatchIssueCredentialResponse, error)

Issues credentials for 1–20 files atomically. Returns *ValidationError if Files is empty or has more than MaxBatchSize (20) entries. Returns a single InsertCalldata for all leaves.

GenerateProof auth POST

GenerateProof(ctx, *GenerateProofRequest) (*GenerateProofResponse, error)

Generates a Groth16 ZK proof for a credential. N and S are validated against the BN254 scalar field before the HTTP call (returns *ValidationError immediately if out of range). Proof generation takes ~600 ms — set WithTimeout ≥ 10s.

PrepareShare auth POST

PrepareShare(ctx, *SharePrepareRequest) (*SharePrepareResponse, error)

Prepares a delegated credential share for a grantee. The middleware issues a fresh credential for the grantee and returns two calldatas: InsertCalldata (grantee leaf) and GrantShareCalldata (share record).

IncomingShares auth GET

IncomingShares(ctx, granteeAddress string) (*IncomingSharesResponse, error)

Lists all credential shares granted to a grantee address, with their current active status (revoked shares have Active: false). Validates address format before the HTTP call.

GetUserPubKey auth GET

GetUserPubKey(ctx, address string) (*UserPubKeyResponse, error)

Returns the on-chain registered secp256k1 public key for a wallet address. Registered: false means the user has not yet called registerPublicKey on-chain.

ResolveEmail auth GET

ResolveEmail(ctx, emailHash string) (*ResolveEmailResponse, error)

Resolves a keccak256(email) hash to a wallet address and public key via the ZKClaimUserRegistry contract. emailHash must be a 0x-prefixed bytes32 hex string.

SetAPIKey

SetAPIKey(apiKey string)

Atomically updates the Bearer token. Safe to call concurrently with in-flight requests. In-flight requests already dispatched continue with the previous key; new requests use the new key.

Security

Private key handling (SEC-01)

Private keys are never logged or included in error messages. CryptoError messages do not embed any key material. GCM authentication failures return a generic message rather than wrapping the underlying Go error (which could expose partial key information in theory).

API key safety (SEC-02)

The Authorization: Bearer <key> header is never included in errors, NetworkError.Cause wrapping, or logs. The header is applied at the last moment before each HTTP dispatch and stripped before constructing any error struct.

Field validation before HTTP (SEC-03)

GenerateProof validates n and s against the BN254 scalar field before making any HTTP call. Sending an out-of-range value to the middleware would return a 400, but the SDK rejects it immediately with *ValidationError to avoid the network round-trip.

Concurrent API key rotation (SEC-04)

SetAPIKey uses sync/atomic.Pointer[string] for the key store. Reading the key and attaching it as a header are done atomically per-request via withHeader (which clones the httpClient immutably). There is no window where a request can see a partially-written key.

ECIES random number generation (SEC-05)

Ephemeral key generation for ECIES uses secp256k1.GeneratePrivateKey(), which uses crypto/rand internally — never math/rand.

DecryptedSecret lifetime

⚠
Limit the lifetime of *DecryptedSecret to the smallest necessary scope. Do not assign it to a struct field, do not pass it to goroutines, and do not include it in logs or error messages. Go does not guarantee memory zeroing when values go out of scope.

ECIES cross-SDK compatibility

The ECIES implementation in github.com/ssanjeewa/go-sdk/crypto is byte-for-byte wire-compatible with:

  • zkp-middleware/internal/crypto/ecies.go (Go middleware)
  • @zkp-system/node-sdk/src/crypto/ecies.ts (Node.js SDK)

A credential encrypted by the middleware can be decrypted by the Go SDK, and vice versa.
