## 4. Security Considerations

### SEC-01 — Private Key Handling

**Area**: `crypto/ecies.go`, `crypto/keys.go`

The SDK handles secp256k1 private keys when encrypting/decrypting secrets. Rules:

- Private keys are **never logged or included in error messages**. `*CryptoError` messages must not embed key material.
- `PrivateKeyToPublicKey` must validate key length (32 bytes / 64 hex chars) before passing to the secp256k1 library — malformed keys cause panics in some secp256k1 implementations.
- `DecryptSecret` must return `*CryptoError` (not wrap the underlying GCM error message verbatim) — GCM errors can leak partial key information in theory.

### SEC-02 — No API Key Logging

**Area**: `http.go`, `client.go`

The `Authorization: Bearer <key>` header must **never** appear in error messages, `NetworkError.Cause` wrapping, or log output. The `httpClient` must strip auth headers before including request details in error structs.

### SEC-03 — Field Element Validation Before HTTP Call

**Area**: `validate.go`, `client.go`

`n` and `s` in `GenerateProofRequest` are validated client-side with `ParseFieldElement` before the HTTP call. Sending an out-of-range value to the middleware returns a 400 but the SDK should reject it immediately with `*ValidationError` to avoid a round-trip. Mirrors the identical check in `sdk/node/src/client.ts:validateProofRequest`.

### SEC-04 — Concurrency Safety of `SetAPIKey`

**Area**: `client.go`

`SetAPIKey` must be safe to call concurrently with in-flight requests. Use `sync/atomic.Pointer[string]` or a `sync.RWMutex` around the API key and httpClient rebuild. The simplest correct approach: store the API key in `atomic.Pointer[string]` and reconstruct the `httpClient` headers on each request from the current key value.

### SEC-05 — Random Number Generation for ECIES

**Area**: `crypto/ecies.go`

Must use `secp256k1.GeneratePrivateKey()` (which uses `crypto/rand` internally) for ephemeral key generation — not `math/rand`. This is already the case in the middleware source being ported; verify it is preserved during the port.

### SEC-06 — Plaintext n/s in Memory

The decrypted `(n, s)` secrets appear in `*DecryptedSecret` as `*big.Int`. Go does not provide guaranteed memory zeroing. Document in godoc that callers should limit the lifetime of `*DecryptedSecret` and avoid logging or serializing the struct. The SDK itself never logs these values.

## 9. Dependency Decisions

### 9.1 `github.com/decred/dcrd/dcrec/secp256k1/v4`

**Why**: The only secp256k1 implementation in the Go ecosystem with:

- `GeneratePrivateKey()` using `crypto/rand`
- `ScalarMultNonConst` for constant-time ECDH (used in middleware)
- Active maintenance and used by go-ethereum itself

**Already used by**: `zkp-middleware/go.mod` — no version conflict risk.

### 9.2 `github.com/ethereum/go-ethereum`

**Why**: `crypto.Keccak256` is needed for scope calculation in the crypto package.
go-ethereum is a large dependency (~5,000 packages), but only `crypto/` sub-package
is used.

**Alternative**: Replace with `golang.org/x/crypto/sha3.NewLegacyKeccak256()` — pure
Go, much smaller. No functional difference for keccak256.

**Decision**: Use `golang.org/x/crypto` to keep the SDK lightweight. The full
`go-ethereum` module is only needed by the middleware server (chain calls), not by
client SDKs.

> **Action**: When implementing Phase 2, use `golang.org/x/crypto/sha3` instead of
> `go-ethereum/crypto` for `Keccak256`. Update `go.mod` accordingly. Add a note in
> `crypto/field.go` explaining why we do NOT use go-ethereum here.

### 9.3 Stdlib only for everything else

`crypto/aes`, `crypto/cipher`, `crypto/rand`, `crypto/sha256`, `encoding/hex`,
`encoding/json`, `math/big`, `net/http`, `regexp`, `sync/atomic` — all stdlib.
